# ドキュメント棚卸しと移行マップ

## 目的

この文書は、既存ドキュメントの責務、重複、移行先を整理し、今後の再編作業で迷わないための基準をまとめたものです。

前提:

- 人間にとって読みやすい構造を優先する
- 要約と参照は許容するが、詳細本文の正本は一箇所にする
- `README.md` は最短入口とし、詳細運用は `docs/` に集約する
- `AGENTS.md` はエージェント固有指示を扱い、人間向けルールの重複記載は減らす
- `docs/agents/skills/*` は今回の棚卸し対象外とする

## 対象外

| パス                   | 理由                                                                 |
| ---------------------- | -------------------------------------------------------------------- |
| `docs/agents/skills/*` | エージェント専用手順であり、今回の通常 docs 再編とは責務が異なるため |

## 重複と問題点

### 1. `README.md` と `docs/development-guide.md`

- 開発開始前のチェック、変更後の検証、完了条件、生成物の扱いが重複している
- 現状は「入口」と「正本」の境界が曖昧で、どちらを更新すべきか判断しづらい
- 今後は `README.md` を要約、`docs/development-guide.md` を正本に分離する

### 2. `docs/note.md` の混在

- `Entry Title` や `downloadUrl` はドメイン仕様寄り
- `Modified or Updated` は文言ルール寄り
- `Error handling (2026-04)` は設計判断と実装方針の記録寄り
- ひとつのメモに用途の異なる内容が混在しているため、検索性と更新判断が悪い

### 3. `AGENTS.md` と人間向け docs の関係

- 環境、規約、テスト方針が docs と重なっている
- エージェント専用指示と一般ルールを分離した方が、保守性が上がる

### 4. `docs/spec.md` の曖昧さ

- 採用ライブラリの一覧としては有用だが、恒久仕様なのか、技術選定メモなのかが曖昧
- 将来的には「技術構成の要約」か「設計判断の入口」のどちらかに寄せる必要がある

## 移行方針

### 入口と正本

| 情報種別                                 | 入口                                    | 正本                                         |
| ---------------------------------------- | --------------------------------------- | -------------------------------------------- |
| プロジェクト概要、導線                   | `README.md`                             | 必要に応じて各 docs                          |
| 開発手順、検証、完了条件、生成物ポリシー | `README.md` の要約                      | `docs/development-guide.md`                  |
| 命名・実装規約                           | `README.md` または `AGENTS.md` から参照 | `docs/naming-convention.md` と今後の規約文書 |
| 技術構成、採用背景                       | `README.md` から参照                    | `docs/spec.md` または再編後の技術文書        |
| 個別の設計判断                           | 必要に応じて関連文書から参照            | `docs/decisions/` 配下の軽量 ADR             |
| エージェント固有指示                     | `AGENTS.md`                             | `AGENTS.md`                                  |

### `docs/note.md` の再配置候補

| 現在の項目                                    | 再配置先の候補                                             | 理由                                                                   |
| --------------------------------------------- | ---------------------------------------------------------- | ---------------------------------------------------------------------- |
| `Entry Title の扱いについて`                  | `docs/domain-behavior.md`                                  | 表示仕様とデータ差異の吸収ルールであり、ドメイン挙動として扱うため     |
| `download link (downloadUrl) の仕様 (2024版)` | `docs/domain-behavior.md`                                  | データ種別ごとの振る舞いを、現在の画面挙動として整理するため           |
| `Modified or Updated`                         | `docs/domain-behavior.md`                                  | UI 表示上の用語使い分けとして扱うため                                   |
| `Error handling (2026-04)`                    | `docs/decisions/2026-04-09-error-handling.md`              | 時点つきの設計判断として判断記録へ移したため                           |

## 実施順

1. `README.md` を入口専用にする
2. `docs/development-guide.md` を運用ルールの正本として整える
3. `docs/naming-convention.md` を規約文書として整理する
4. `docs/decisions/` を追加し、設計判断の受け皿を用意する
5. `docs/spec.md` の責務を再定義する
6. `docs/note.md` の内容を各所へ移し、最終的に廃止する
7. `AGENTS.md` を docs 参照型に整理する
8. 役割を失った文書の削除と導線確認を行う
