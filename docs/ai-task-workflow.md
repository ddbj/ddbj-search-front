# AI 改修ワークフロー

## この文書の役割

この文書は、人間が明示的に起動する AI 改修ワークフローをまとめる正本です。

- 複数 issue に分かれる改修を、AI と人間で整理しながら進める
- 仕様、進行、記録、実装の置き場所を混線させない
- 計画と実装の責務を分け、issue ごとにコンテキストを分離する

この文書は、AI が常時自動で採用する標準フローではありません。
人間が「この流れで進める」と明示したときだけ使います。

この文書でいうエージェントは、Codex 上で起動する作業単位を指します。

## このフローを使う前提

次のような作業で使います。

- 仕様整理が必要で、そのまま実装に入るには前提が多い
- 複数の Linear issue に分けて進めたほうが整理しやすい
- 案件全体を Linear Project で管理したい
- 親エージェントは全体進行管理に徹し、実務は専用エージェントへ分けたい

Notion はこのフローの前提ではありません。
必要なら補助的に使ってもよいですが、正本や必須導線としては扱いません。

## 全体の流れ

1. 人間がこのフローを使うことを明示し、Linear Project を用意して指定する
2. 計画専用エージェントが、既存資料を確認したうえで、ユーザーへ詳細をインタビューする
3. 計画専用エージェントが、論点整理、Project Description 起案、issue案の分割を行う
4. 計画専用エージェントが、issue案の一覧、大まかな順序、各 issue案の推奨 Model / Reasoning をユーザーへ提示し、承認を得る
5. 計画専用エージェントが、承認された issue案を issue や child issue として起票し、Project Description と依存関係を反映する
6. 実装専用エージェントが、担当 issue ごとに詳細計画、実装、検証、記録更新を行う
7. 実装前計画で再分割が必要になった場合は、そのエージェントが計画専用エージェント扱いに移行し、同じ手順で child issue案を整理して承認を得る
8. 承認された child issue案を child issue として起票し、実装は新しく起動した別エージェントへ移す
9. issue ごとの完了後に必要な review や確認を済ませる
10. Project 全体の最終ふりかえりを行い、人間が Done を判断する

原則として進行は直列です。
必要な場合だけ並列化を検討します。

## 仕様の扱い

- 仕様や判断は、まず会話で整理する
- 恒久的に残すべき判断があれば、プロジェクト完了時に必要な文書へ反映する
- 進行中の仕様や判断を docs に先回りして過剰に書き込まない
- 仕様そのものを Project や issue の進行メモへ過剰に埋め込まない

## 媒体ごとの責務

### Linear Project

- 案件全体の親として扱う
- Project Description を、案件全体の現在の正本として扱う
- Project コメントを、判断理由、差し戻し、補足を残す流れるログとして扱う
- Project コメントは、エージェントごとに失われやすい暗黙知を可視化するための判断ログとしても使う
- Project Update を、大きな進捗や判断変更の節目にだけ使う
- 長期プロジェクトで Project コメントの判断ログが追いにくくなった場合は、Project Update を中間要約にも使ってよい

### Linear Issue

- issue ごとの実行単位、状態、担当、完了判断の正本として扱う
- issue コメントを、完了メモ、review result、次に参照すべき内容を残す最小記録として扱う
- 標準構造は `Project > issue` とし、さらに細分化が必要な場合だけ `child issue` を使う

### docs

- プロジェクト完了時に、将来も参照価値がある判断だけを昇格する
- 最終ふりかえりの結果、何も昇格しない場合もある
- docs 化が重い場合だけ、別 issue に分けることを検討する

### Git

- commit は Done 条件ではない
- 必要に応じて複数回 commit してよい
- 完了判定は commit ではなく Linear の状態管理で行う

## Project Description の扱い

Project Description は、案件全体の現在の骨格だけを持ちます。
進行ログとして使わず、最新状態の要約だけを残します。

含める内容は、少なくとも次のとおりです。

- 目的
- 現在の構造
- 主要依存 / 大枠の進め方
- 補足
  - 任意。複数 issue に効く前提だけを書く

Project Description は各 issue 担当者が更新してよいですが、更新したら必ず Project コメントに判断理由を残します。

## 親エージェントの責務

親エージェントは、受付と起動管理だけを担当します。

- 案件全体の現在地を把握する
- 次に起動する計画専用エージェント / 実装専用エージェントを選ぶ
- 人間へ現在地と次の動きを共有する

親エージェントが次の起動判断に迷う場合は、人間に委ねます。
構造変更が必要な場合は、計画専用エージェントを起動します。

親エージェントは、次のことを持ちません。

- 全体計画の具体化
- Project Description の初期作成
- issue 起票
- 各 issue の詳細計画
- 実装や検証
- 各 issue の commit 判断

## 計画専用エージェントの責務

計画専用エージェントは必須です。
案件の最初に起動し、全体の進め方を設計します。

- 計画品質を優先し、高性能モデルと高めの Reasoning を選ぶ
- 既存資料を確認したうえで、必ずユーザーへ詳細をインタビューする
- 案件全体の目的を整理する
- Project Description を起案する
- issue案を細かく分割する
- 主要依存 / 大枠の進め方を整理する
- 各 issue案の難易度を見て、推奨 Model / Reasoning を整理する
- issue案の一覧、大まかな順序、各 issue案の推奨 Model / Reasoning をユーザーへ提示し、承認を得る
- 承認された issue案を issue や child issue として起票する
- 再分割が必要になったときに再計画する

計画専用エージェントは、次のことを持ちません。

- Project の初期作成
- 実装
- 検証
- 個別 issue の commit

## 実装専用エージェントの責務

実装専用エージェントは、自分の issue を閉じるところまで担当します。

- 着手前に issue の詳細計画を作る
- 実装と検証を行う
- Linear の状態更新とコメント記録を行う
- 必要に応じて Project Description を更新する
- 必要に応じて commit を作る

実装前の詳細計画で再分割が必要になった場合は、そのエージェントが計画専用エージェント扱いに移行します。
その場合も、既存資料確認、ユーザーインタビュー、child issue案の整理、推奨 Model / Reasoning の検討、承認取得を行います。
承認された child issue案の起票と詳細実装計画の整理まで行って手を止め、実装は新しく起動した別エージェントへ移します。
構造変更を行った場合は、その内容を親エージェントへ共有し、次に起動するエージェントの判断を委ねます。

## 進行ルール

- issue は細かく切る
- 基本粒度は `1 issue = independently verifiable な最小単位` とする
- この粒度は厳密な機械判定ではなく、計画専用エージェントが確認シナリオと handoff のしやすさを見て判断する
- UI に影響する変更は、人間が確認しやすい単位で切る
- UI 変更は、原則として 1 つの確認シナリオで完結する単位を優先する
- 粒度に迷いがある場合は、無理に細分化せず人間に委ねてよい
- 後続 issue が前の issue の成果物に依存する場合は、主要依存 / 大枠の進め方を見直す
- 推奨 Model / Reasoning は各 issue案の作成時に、人間と AI が難易度を見ながら議論して決める
- この文書にはモデル名や固定の Reasoning レベルは書かない
- 起動したエージェントが失敗したり、長時間停止したり、前提が崩れたりした場合は、そのまま進めず、どこで止まったか、何が不足しているか、再開に必要な判断を短く残す
- 外部ツール利用時の待ち方や中断時の扱いは `docs/external-tool-operation.md` を参照する

## 再分割の扱い

issue をさらに分ける必要がある場合は、child issue を使います。

- 元 issue は、それ以降は親 issue として扱う
- 実装は child issue に完全に移す
- 親 issue は、子 issue がすべて完了し、束ねコメントを残した時点で完了とする

## Review の扱い

UI に影響する issue は、実装完了と human review 完了を分けて扱います。

- review request は会話上で行ってよい
- review result だけを issue コメントに最小限残す
- UI に影響する issue は、human review 完了後に Done とする
- review 待ちの間でも、review 結果に依存しない後続 issue は先行着手してよい
- 次の着手可否に迷う場合は、親エージェントが人間に委ねる

状態名そのものは固定ルールにしません。
必要なら Linear の `In Review` のような review 可能状態を使います。

## 完了時の締め方

issue は、その種別に必要な検証と記録更新が済んだ時点で完了に進めます。

補足:

- code change がある場合、commit は必要に応じて途中でも行ってよい
- issue 完了は commit ではなく Linear の状態管理で判定する
- UI に影響する issue は human review 完了後に Done とする

Project では、最終ふりかえりを行います。
その要約は Project Update に残し、Project を Done にするかどうかは人間が判断します。

この締め方は AI 改修フロー特有の補足であり、一般の開発完了条件は `docs/development-guide.md` を参照します。

## 記録ルール

媒体ごとの役割分担は、上の `媒体ごとの責務` を正本とします。

issue 完了時コメントは結果の要約として使います。
後続エージェントが再現しにくい判断経緯、試して見送った案、複数 issue に効く制約は、暗黙知の可視化のために Project コメントへ判断ログとして残します。

issue を閉じる前は、少なくとも次を確認します。

- Linear の状態更新と必要なコメントが反映されている
- Project Description を更新した場合は、Project コメントに判断理由が残っている
- UI に影響する issue では human review の結果を確認できる

## テンプレート

### Project Description

```markdown
## 目的

## 現在の構造

## 主要依存 / 大枠の進め方

## 補足
```

`補足` は任意です。
複数 issue に効く前提だけを残します。

### 計画専用エージェントが最初にまとめる項目

- 対象 Project
- ユーザーインタビューで確認した前提
- 案件の目的
- 現在の構造
- 主要依存 / 大枠の進め方
- issue案一覧
- 順序案
- 各 issue案の推奨 Model / Reasoning

### issue Description

各 issue には、少なくとも次を含めます。

- 目的
- done condition
- 推奨 Model / Reasoning
- review 観点
  - 必要な場合だけ書く

推奨 Model / Reasoning は原則としてその issue の開始時に従います。
ただし、着手時に見直しが必要なら、人間確認のうえで変更してよいです。

### 実装専用エージェントの開始時に明示する項目

- 対象 issue
- issue の目的
- 詳細 done condition
- 再分割が必要かどうか

### issue 完了時コメント

実装専用エージェントの終了時は、少なくとも次を短く残します。

```markdown
- 何を終えたか:
- 次に参照すべきこと:
- 残件 / 前提:
```

`次に参照すべきこと` と `残件 / 前提` は、不要なら「なし」でよいです。

## 見直し条件

- 親エージェント、計画専用エージェント、実装専用エージェントの役割分担が変わったとき
- Project / issue / docs の役割分担が変わったとき
- Project Description、Project コメント、Project Update の使い分けが変わったとき
- review 条件や確認方法が変わったとき
- この運用より適切な共通フローが別文書に整理されたとき
