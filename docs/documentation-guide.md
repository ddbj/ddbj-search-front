# ドキュメントガイド

## この文書の役割

この文書では、リポジトリ内のドキュメントを書くときの基本方針を定める。

- 人間にもAIにも読みやすい文章を書く
- 文書ごとの役割を分ける
- 更新時に重複や役割の衝突を増やさない

## 基本方針

- まず、その文書が何のための文書かを明確にする
- 1つの文書には役割を絞り、別の役割の説明を抱え込みすぎない
- 必要以上に重複して説明せず、詳細は適切な文書へ案内する
- 読み手が次に何を見ればよいか分かるように、関連文書への導線を残す

## 読みやすい文章の方針

- 1文を長くしすぎない
- 箇条書きで済む内容は、無理に段落へ押し込まない
- 主語や対象が曖昧な書き方を避ける
- 「何をする文書か」「何は扱わないか」を必要に応じて書く
- 前提知識がなくても追いやすい順で情報を並べる

## 更新時の意識

- 文書を追加・更新するときは、既存の基準文書と役割が重ならないか確認する
- 既存文書の一部を別文書へ分けた場合は、[docs/index.md](./index.md) などの入口側の導線も更新する
- 一時的な判断と恒久ルールを同じ場所に混在させない

## 作業メモの置き方

- タスクを進めるための一時メモは、作業中だけ必要な情報として扱う
- Gitに残るとノイズになりやすいので、原則としてリポジトリ内には置かない
- 置き場所はNotionやConfluenceなどの外部ツールを優先する
- どのツールを使うべきか、どこに残すべきか判断できない場合は、作業前にユーザーへ確認する
- その場限りのメモと、将来も参照する文書は分けて扱う

## 新しい文書を作るとき

- まず、既存の文書に追記できないか確認する
- 恒久的なルールや手順は `docs/` 直下に置く
- 特定の時点での設計判断は [`docs/decisions/`](./decisions/README.md) に置く
- 新しい文書を追加したら [docs/index.md](./index.md) にも導線を追加する
- ファイル名は役割が分かる短い英語にする（例: `code-style.md`, `testing-policy.md`）

## 文書を廃止・統合するとき

- 内容を移行先へ反映したうえで、元のファイルを削除する
- [docs/index.md](./index.md) や他の文書からのリンクも合わせて更新する
- 移行先が複数にまたがる場合は、どこに何を移したかをコミットメッセージに残す

## ドキュメント変更時の扱い

- ドキュメント変更のみなら、原則として `test` / `lint` / `type-check` は実行しない
- ただし、設定ファイル、ビルド挙動、Storybook、生成物ポリシーのように実行結果へ影響する説明を変えた場合は、必要な範囲で検証する
- 実行した検証と、省略した検証の理由は完了報告時に共有する
